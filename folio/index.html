WHY

왜 만들었나

AI 코딩 툴을 활용해, PM이 어디까지 직접 제품을 만들 수 있는지 확인해보고 싶었습니다.

MOTIVATION
직접 만들어봐야 더 정확하게 기획할 수 있다
PM이 모든 기능을 직접 개발해야 한다고 생각하지는 않습니다. 다만 구현 과정을 어느 정도 이해해야, 디자인팀·개발팀과 논의할 때 가능성과 제약을 더 정확하게 판단할 수 있다고 생각했습니다. Folio는 제가 직접 구조를 짜고, 기능을 만들고, 문제를 고치면서 만드는 과정을 아는 PM이 되기 위해 시작한 프로젝트입니다.
구현 과정을 알아야 더 정확하게 기획할 수 있다
HYPOTHESIS
캐릭터 대화는 LLM을 가장 직관적으로 보여줄 수 있다
LLM 기반 서비스는 많지만, 사용자가 체감하기 쉬운 형태는 결국 대화라고 생각했습니다. 그중에서도 캐릭터 챗봇은 말투, 기억, 관계성, 응답 일관성 같은 요소가 중요하기 때문에 LLM의 장점과 한계가 가장 잘 드러납니다. Folio는 이 가설을 실제 제품 형태로 검증해보기 위한 시도였습니다.
캐릭터는 LLM의 가능성과 한계를 가장 빠르게 드러낸다
CONTEXT
AI 코딩 툴이 보편화되면 PM의 역할도 달라진다
AI 코딩 툴 덕분에 PM도 아이디어를 문서로만 남기지 않고, 직접 프로토타입을 만들 수 있는 환경이 되었습니다. 그래서 앞으로의 PM은 무엇을 만들지를 정하는 것뿐 아니라, 문제를 어떤 구조로 쪼개고, AI에게 어떻게 요청하고, 결과물을 어떻게 검증할지까지 판단할 수 있어야 한다고 생각했습니다.
AI를 쓰는 PM은 아이디어보다 구조를 더 잘 다뤄야 한다
HOW

어떻게 만들었나

캐릭터 설계, 프롬프트 구조, 어드민 기능, 보안과 배포까지 직접 설계하고 구현했습니다.

CHARACTER DESIGN
캐릭터가 흔들리지 않게 설계하기
캐릭터 챗봇은 기능보다 대화 품질이 먼저 체감되는 서비스라고 생각했습니다. 그래서 각 캐릭터마다 페르소나, 말투, 관계성, 세계관 정보를 먼저 정리했고, 대화 중에도 캐릭터가 같은 기준으로 반응할 수 있도록 프롬프트에 반영했습니다.
캐릭터가 일관되어야 대화가 서비스처럼 느껴진다
PROMPT ENGINEERING
프롬프트를 역할별로 나누기
처음에는 하나의 프롬프트에 모든 정보를 넣으면 된다고 생각했지만, 대화가 길어질수록 캐릭터 설정이 흐려지고 응답 기준이 흔들렸습니다. 그래서 시스템 프롬프트, 대화 메모리, 가드레일을 분리해 캐릭터 설정 유지와 부적절한 응답 방지를 따로 관리했습니다.
프롬프트도 기능처럼 구조가 필요하다
VIBE CODING
AI 코딩 툴로 제품 흐름까지 직접 만들기
Cursor와 Claude를 활용해 캐릭터 채팅 화면, 어드민, 보안 처리, 배포 흐름까지 직접 만들었습니다. 코드를 모두 손으로 짜는 것보다 중요한 것은, 어떤 기능이 필요하고 어떤 순서로 만들어야 하는지 판단하는 일이었습니다. 이 과정에서 PM도 AI 도구를 활용하면 아이디어를 실제 제품에 가까운 형태로 검증할 수 있다는 것을 확인했습니다.
AI 도구를 써도 결국 중요한 건 무엇을 만들지 판단하는 일이다

FEATURES

주요 기능

FEATURE 01
캐릭터별 페르소나 챗
페르소나, 말투, 세계관이 다른 AI 캐릭터와 1:1로 대화할 수 있습니다. 대화 중 캐릭터 설정이 흔들리지 않도록 응답 기준을 프롬프트에 반영했습니다.
FEATURE 02
대화 메모리 관리
이전 대화의 맥락을 저장하고 다시 불러와, 캐릭터가 사용자를 기억하는 듯한 경험을 만들었습니다. 단발성 챗봇이 아니라 관계가 이어지는 대화를 목표로 설계했습니다.
FEATURE 03
가드레일 설계
부적절한 응답을 줄이기 위해 응답 기준과 제한 조건을 별도로 관리했습니다. 안전성을 확보하면서도 캐릭터의 말투와 설정이 깨지지 않도록 설계했습니다.
FEATURE 04
AI 빌더 : 대화형 캐릭터 제작
폼 입력이 아니라 AI와의 대화를 통해 캐릭터를 만들 수 있습니다. 사용자의 막연한 아이디어를 페르소나, 말투, 행동 규칙으로 정리해 캐릭터 설정에 반영하는 기능입니다.
KEY DECISIONS

PM으로서 내린 의사결정

기획자가 구축까지 하면서 만나는 의사결정의 순간들. 감이 아닌 근거로 결정했습니다.

DECISION 01
MODEL SELECTION
Gemini 3.1 Pro를 기본값으로 선택한 이유
벤치마크 수치가 아닌 실제 캐릭터 대화 맥락에서 직접 6개 모델을 비교 검증했습니다.
벤치마크 수치는 우리 맥락과 다르고, PM이 직접 검증하지 않으면 의사결정이 추측이 됩니다. Anthropic 3개(Sonnet 4.6 / Opus 4.6 / Haiku 4.5)와 Google 3개(Gemini 2.5 Flash·Pro / 3.1 Pro)를 동일 캐릭터·동일 시나리오로 테스트했습니다. 캐릭터 일관성·공감 품질·응답 속도·비용을 종합해 Gemini 3.1 Pro를 기본값으로 결정했습니다.
DECISION 02
PROMPT ARCHITECTURE
3층 프롬프트 구조를 채택한 이유
단일 프롬프트로는 캐릭터 일관성과 가드레일을 동시에 처리할 수 없었습니다.
시스템 프롬프트(캐릭터 정의) · 대화 메모리(컨텍스트 유지) · 가드레일(안전성)을 분리된 레이어로 설계했습니다. 이 구조가 없으면 캐릭터가 대화 중 일관성을 잃거나, 가드레일이 캐릭터 톤과 충돌하는 문제가 발생합니다.
DECISION 03
DESIGN SYSTEM
개인 프로젝트에도 디자인 시스템을 도입한 이유
임시 스타일링을 반복하다 보면 컴포넌트 간 일관성이 깨지고 수정 비용이 급격히 올라갑니다.
색상 토큰 · 타이포그래피 스케일 · 컴포넌트 카탈로그를 처음부터 정의했습니다. 개인 프로젝트라도 시스템 없이 시작하면 나중에 전체 리팩터링이 필요합니다. 이 결정이 유지보수 속도를 일정하게 유지해줍니다.
DECISION 04
SECURITY ARCHITECTURE
세션 소유권 검증을 직접 설계한 이유
라이브 운영 중인 프로덕트는 PM도 보안 사고 책임자입니다.
비로그인 게스트의 세션이 다른 게스트와 섞이거나, 로그인 유저가 타인 세션에 접근하는 시나리오를 차단하기 위해 verifyOwnership 헬퍼를 신설했습니다. 로그인 유저는 user_id 일치, 게스트는 guest_id 일치 + user_id IS NULL 검증을 적용하고, 모든 세션 라우트(GET/PUT)에 소유권 체크를 박았습니다. 49개 테스트로 404/403/200 시나리오를 검증했습니다.
DECISION 05
BUILDER UX
캐릭터 제작에 AI 빌더를 도입한 이유
빈 폼을 채우는 방식은 추상적 설정으로 끝나기 쉽습니다.
기존 캐릭터 AI 플랫폼은 빈 양식을 채우는 폼 기반 제작이라 "다정하고 차가운" 같은 추상 설정에 머물기 쉽습니다. AI 에이전트가 추가 질문으로 구체화를 유도하면 비제작자도 구체적 행동 규칙 기반 캐릭터를 만들 수 있다고 봤습니다. /builder/chat (대화형)과 /builder/manual (폼) 양쪽을 모두 제공해 제작자 선택권을 보장했습니다.
