WHY

왜 만들었나

회사 업무가 아닌 가비님 본인의 의지로 시작한 프로덕트. 그 이유를 두 가지로 정리했습니다.

MOTIVATION
PM이 직접 만들 수 있어야 진짜 기획이 가능하다
구축 가능한 PM이 되어야 디자인팀·개발팀과 같은 언어로 논의할 수 있고, 제품의 가능성과 제약을 처음부터 함께 사고할 수 있습니다. 기획자이기 전에 만드는 사람이 되고 싶었습니다.
"AI 시대에 PM은 무엇을 만들 수 있어야 하는가"
HYPOTHESIS
캐릭터 기반 대화는 LLM의 공감 능력을 극대화한다
일반 챗봇이 아니라 페르소나가 있는 캐릭터와의 대화에서 LLM의 자연스러운 공감과 일관성이 더 강하게 드러난다는 가설. Folio는 이 가설의 라이브 검증 실험입니다.
"캐릭터 페르소나가 LLM 일관성을 높인다"
CONTEXT
AI 코딩 툴 보편화 시대, PM의 역할이 달라진다
AI를 잘 활용하는 PM이라면 아이디어를 던지는 사람이 아니라, 어떤 프롬프트 구조로 문제를 정의할지, 어떤 방식으로 LLM을 연결할지 직접 판단할 수 있어야 합니다.
"AI 시대 PM에게 요구되는 역량이 달라졌다"
HOW

어떻게 만들었나

캐릭터 설계부터 프롬프트 아키텍처, 보안까지 1인으로 직접 설계한 방식.

CHARACTER DESIGN
페르소나 · 말투 · 세계관
캐릭터별 고유한 페르소나, 말투, 세계관을 정의. 대화 중 일관성을 유지하기 위한 설계 원칙을 캐릭터마다 별도로 작성했습니다.
"캐릭터 일관성이 UX 품질을 좌우한다"
PROMPT ENGINEERING
3층 프롬프트 아키텍처
시스템 프롬프트 · 대화 메모리 · 가드레일의 3층 구조로 설계. 캐릭터 일관성 유지와 부적절 응답 방지를 동시에 처리합니다.
"구조가 없으면 캐릭터가 무너진다"
VIBE CODING
AI 보조 풀스택 구축
Cursor, Claude, 바이브 코딩 워크플로우를 활용해 기획자가 직접 어드민·보안·배포까지 처리. 코드보다 판단과 설계에 집중하는 방식입니다.
"도구가 달라져도 PM의 판단은 그대로"

KEY DECISIONS

PM으로서 내린 의사결정

기획자가 구축까지 하면서 만나는 의사결정의 순간들. 감이 아닌 근거로 결정했습니다.

DECISION 01 · MODEL SELECTION
Gemini 3.1 Pro를 기본값으로 선택한 이유
벤치마크 수치가 아닌 실제 캐릭터 대화 맥락에서 직접 6개 모델을 비교 검증했습니다.
벤치마크 수치는 우리 맥락과 다르고, PM이 직접 검증하지 않으면 의사결정이 추측이 됩니다. Anthropic 3개(Sonnet 4.6 / Opus 4.6 / Haiku 4.5)와 Google 3개(Gemini 2.5 Flash·Pro / 3.1 Pro)를 동일 캐릭터·동일 시나리오로 테스트했습니다. 캐릭터 일관성·공감 품질·응답 속도·비용을 종합해 Gemini 3.1 Pro를 기본값으로 결정했습니다.
DECISION 02 · PROMPT ARCHITECTURE
3층 프롬프트 구조를 채택한 이유
단일 프롬프트로는 캐릭터 일관성과 가드레일을 동시에 처리할 수 없었습니다.
시스템 프롬프트(캐릭터 정의) · 대화 메모리(컨텍스트 유지) · 가드레일(안전성)을 분리된 레이어로 설계했습니다. 이 구조가 없으면 캐릭터가 대화 중 일관성을 잃거나, 가드레일이 캐릭터 톤과 충돌하는 문제가 발생합니다.
DECISION 03 · DESIGN SYSTEM
개인 프로젝트에도 디자인 시스템을 도입한 이유
임시 스타일링을 반복하다 보면 컴포넌트 간 일관성이 깨지고 수정 비용이 급격히 올라갑니다.
색상 토큰 · 타이포그래피 스케일 · 컴포넌트 카탈로그를 처음부터 정의했습니다. 개인 프로젝트라도 시스템 없이 시작하면 나중에 전체 리팩터링이 필요합니다. 이 결정이 유지보수 속도를 일정하게 유지해줍니다.

WHY ENGINEERING RIGOR

왜 PM이 여기까지 했는가

PM IDENTITY
"디자인을 이해해주는 PM, 개발을 이해해주는 PM이 되고 싶었습니다"
항해 백엔드 수료 경험으로 프로덕션 기준을 알았기 때문에, PM이지만 시스템 안정성까지 직접 구현하는 선택이 자연스러웠습니다.
VIBE CODING TRAP
"LLM 제품의 가장 큰 리스크는 부가적 요소를 놓쳐 발생하는 애프터매스입니다"
안정성, 보안, 유지보수성 — 빠른 개발이 가능한 만큼 이런 부가적 요소를 시스템적으로 통제하지 않으면 운영에서 무너집니다.
HARNESS RATIONALE
"Claude Code의 단기기억 특성을 시스템으로 보완해야 합니다"
7개 마크다운과 Changelog는 토큰 절약 + 컨벤션 유지 + 유지보수성 보장을 위한 의도적 선택이었습니다. 작업 맥락이 사라지면 안정성이 무너지니까요.

LEARNED

만들면서 배운 것

LEARNED 01
구축 능력이 있는 PM은 디자인팀·개발팀과 같은 언어로 논의할 수 있다
Folio를 직접 만들면서 구현 가능성·비용·구조적 제약을 함께 보면서 판단하는 능력이 생겼습니다. 이것이 회사 PM 업무에서도 더 현실적인 의사결정의 기반이 되었습니다.
LEARNED 02
LLM 프롬프트 엔지니어링은 PM의 새로운 핵심 스킬이다
어떤 프롬프트 구조로 문제를 정의할지, 어떤 방식으로 LLM을 연결할지, 어떤 흐름으로 사용자 경험을 설계할지까지 직접 판단할 수 있어야 AI 시대의 PM입니다. Folio는 그 감각을 검증한 실험입니다.
LEARNED 03
디자인 시스템은 개인 프로젝트에서도 효과를 발휘한다
처음부터 토큰과 컴포넌트를 정의해두면 기능 추가 속도가 유지됩니다. 시스템 없이 시작했다면 지금쯤 전체 리팩터링이 필요했을 것입니다.
