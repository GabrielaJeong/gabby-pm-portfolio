CONTEXT
03

두 서비스가 하나의 인프라를 공유하는 구조

메타브와 메타브 스튜디오는 별개의 프로덕트지만, 같은 서버를 쓰며 비용을 함께 짊어졌습니다.

메타브는 회원 80만, 월 30만 UV 규모의 B2C 플랫폼으로, 다양한 심리/성격 콘텐츠를 무료로 제공합니다. 트래픽은 컸지만 직접적인 결제 모델이 없었습니다.

메타브 스튜디오는 메타브에서 사용하는 콘텐츠 제작 SaaS로, 외부 제작자도 결제하면 사용할 수 있습니다. 4단계 요금제(Free/Plus/Pro/Enterprise)를 운영했지만 저가 요금제 중심으로 결제가 발생해 인프라 비용을 회수하지 못했습니다.

두 서비스는 동일한 서버 인프라를 사용하고, 사용자가 늘어날수록 손실이 커지는 구조였습니다.

B2C 사용자가 늘어도 직접 결제는 없고,
SaaS 결제는 저가에 쏠려 인프라 비용을 회수하지 못했습니다.
PROBLEM
04

한 번의 인터뷰가 세 가지 의사결정의 근거가 되었습니다

가격, 상품 혼선, 기능 가치 — 인터뷰 응답에서 서로 다른 의사결정의 단서를 동시에 발견했습니다.

무료 사용자(메타브)는 비용만 늘렸고, 유료 사용자(스튜디오)는 저가 요금제에 결제가 쏠려 회수가 어려웠습니다. 가격을 올리면 이탈할 가능성이 컸기 때문에, 먼저 사용자가 어디까지 지불할 의향이 있는지 측정이 필요했습니다.

CASE 02 · STRATEGY

혼선이 만든 비용을 UX로 풀다

CS 15-17%, 환불 반복, 플랫폼 이탈 — 같은 원인의 다른 증상

STEP 01 · OBSERVATION
문제 정의
동일 유형 CS가 전체 문의의 15-17%를 반복적으로 차지. 환불 요청도 같은 사유. 플랫폼 이탈까지 이어짐.
"표면 증상이 아니라 구조 문제"
STEP 02 · HYPOTHESIS
정보의 부재가 아닌 노출 시점의 문제
홈페이지에 상품 설명은 이미 명시되어 있었지만, 사용자가 결제 시점에 정보를 다시 찾지 않음. 정보 자체가 아니라 노출 타이밍이 핵심.
"유저는 정보를 찾지 않고, 마주친다"
STEP 03 · APPROACH
사용자 여정 전반에 안내 삽입
진입/결제/사용 각 시점에 두 상품의 차이가 자연스럽게 드러나도록 UX 재설계. 개발 신규 기능 없이 노출 구조만 변경.
"리소스 최소화, 임팩트 최대화"
CASE 02 · EXECUTION

구매 전·중·후 모든 여정에 안내를 삽입

기능 추가 없이 UX 노출 구조만 4곳 수정

ACTION 01
유저 목적 기반 진입 경로 분리
사용자 목적(콘텐츠 제작 vs 운영)에 따라 자연스럽게 분기되는 플로우 설계. 콘텐츠 제작자와 플랫폼 운영자가 서로 다른 경로로 진입.
진입 단계에서 상품 혼선 방지
ACTION 02
로그인 이후 랜딩 구조 개편
멤버십 페이지 이탈률이 전체 홈페이지 이탈률의 30%를 차지하던 문제 해결. 템플릿 상품을 인지할 수 있는 '템플릿 사용하기' 페이지로 이동.
멤버십 페이지 이탈률 회복
ACTION 03
상품 정보 노출 시점 재설계
스튜디오 가이드 전면 개편. CS가 가장 많이 접수되던 페이지에서 상품 차이를 직관적으로 설명. 가이드 자체가 사전 안내 채널로 기능.
가이드 페이지가 안내 채널로 작동
ACTION 04
CS 채널 빠른 응답 템플릿 구축
동일 유형 CS에 대한 응답 템플릿 사전 작성. 운영자가 복사·붙여넣기만으로 일관된 응답 제공 가능. 운영 부담 감소.
CS 응답 시간 단축, 운영 피로 감소

CASE 02 · REFLECTION

이 케이스에서 배운 것

LEARNED 01
유저의 이해 부족이 아니라 플로우 설계의 문제였다
CS가 반복되면 "안내 부족"으로 보이지만, 실제로는 정보 노출 타이밍이 잘못된 경우가 많습니다. 유저는 정보를 찾으러 가는 게 아니라 마주치는 위치에서만 인식합니다.
LEARNED 02
정보의 내용보다 노출 타이밍이 UX를 결정한다
같은 정보라도 사용자가 정보를 필요로 하는 시점에 노출되어야 의미가 있습니다. 정보의 정확성보다 정보의 위치가 UX의 본질입니다.
GOOD
  • 데이터 기반 원인 추적 (CS 로그 + GA 이벤트 + 채널 유입경로)
  • 개발 리소스 없이 UX 노출만으로 임팩트 달성
LIMIT
  • A/B 테스트 없이 일괄 적용 (변경 효과의 정밀 측정 한계)
  • 사용자 의도 분기 플로우의 장기적 검증 부재

"구조를 바꾸면 인력 의존을 벗어날 수 있다"
CASE B · EXECUTION

MVP는 무엇을 뺄지 결정하는 작업

3개 핵심 설계 원칙 + 의도적인 MVP 범위 축소

PRINCIPLE 01
직관적 화면 구성
심플한 유저 친화적 화면 + 기능. 에디터 학습 곡선 최소화로 진입장벽 완화.
PRINCIPLE 02
레이어 개념 활용
피그마/포토샵 등에서 사용되는 '레이어' 개념 차용. 유저가 다양한 레이어 구성으로 창의적 작업 가능하게 설계.
PRINCIPLE 03
웹 편집기 / 앱 플레이어 합일성
제작자(웹 편집기)와 플레이어(앱) 양쪽의 기능 구성을 일치시켜 일관된 UX 제공.
MVP SCOPE DECISION
"무엇을 뺄지 결정"
초기 기획안에는 텍스트 삽입 기능과 편집기 내 기본 레이어 노출이 포함되어 있었지만, MVP 범위에서는 의도적으로 제외했습니다. 리소스 제약 안에서 핵심 가설을 가장 빠르게 검증하기 위해, 부가 기능 대신 본질 기능에 집중하는 결정이었습니다.

CASE 03 · META REFLECTION

두 번의 0→1에서 배운 것

다른 도메인의 두 기능, 같은 사고방식을 적용했습니다

LEARNED 01
MVP는 무엇을 만들지가 아니라 무엇을 뺄지 결정하는 것
0→1 기획에서 욕심을 내면 출시 자체가 미뤄집니다. 핵심 가설을 가장 빠르게 검증할 수 있는 범위까지 의도적으로 줄이는 것이 MVP의 본질입니다. 두 케이스 모두 같은 원칙을 적용했습니다.
LEARNED 02
시장 시그널이 검증된 순간이 최적의 출시 타이밍이다
닉네임 기능은 800만 참여 콘텐츠가 보낸 신호에서, 이미지커스텀은 픽크루의 검증된 수요에서 출발했습니다. 0→1 기획은 수요를 만드는 작업이 아니라 검증된 수요를 발견하고 번역하는 작업입니다.
GOOD
  • 시장 시그널 기반 가설 수립 → 출시 후 즉시 결제/판매 검증
  • 기능 자체가 결제 동기로 작동하는 구조 설계 (닉네임 → Pro)
LIMIT
  • 이미지커스텀 — GTM 전략이 출시 시점부터 함께 설계되어야 했음
  • 두 케이스 모두 장기 retention 검증까지는 이르지 못함

META SKILL

모든 케이스의 토대 — 이해관계자 조율

PM 직무가 결국 의사결정을 통과시키는 일이라는 것을, 3개 케이스 모두에서 배웠습니다

메타브에서 다룬 모든 케이스의 의사결정은 결국 사람을 통과해야 했습니다. 6배 가격 인상은 영업·고객 운영팀의 우려를 통과해야 했고, 엔터프라이즈 사업 전환은 콘텐츠팀 리소스 재배분 협의가 필요했고, 이미지커스텀 MVP 범위 축소는 기획자의 욕심과 부딪혔습니다.

저는 이해관계자 조율을 별도 케이스로 분리하지 않았습니다. 3개 케이스 모두에서 일어난 일이었기 때문입니다. 어떤 의사결정이든, 그 결정이 실행에 옮겨지려면 동의를 모아야 한다는 것을 매일 배웠습니다.

조율 방식은 케이스마다 달랐습니다. WTP 인터뷰 데이터로 가격 인상 근거를 확보한 것도 조율이었고, CS 데이터 트렌드로 UX 개편의 우선순위를 설득한 것도 조율이었고, 픽크루 레퍼런스와 납품 리드타임 수치로 MVP 범위를 좁힌 것도 조율이었습니다. 데이터와 사례는 결국 사람을 설득하기 위한 언어였습니다.

WITH TEAMS
부서 간 리소스 협의
개발/디자인/콘텐츠/영업 팀과 우선순위 충돌이 있을 때마다 직접 협의해 합의안을 만들었습니다.
WITH CLIENTS
