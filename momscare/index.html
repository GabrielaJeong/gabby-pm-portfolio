CONTEXT
03

맘스케어는 어떤 서비스인가

양육자의 멘탈 헬스를 케어하는 CSR 플랫폼. 후원사 2곳과 운영사가 역할을 나눠 운영합니다.

맘스케어는 초록우산 어린이재단과 한화생명이 후원하고, 더에스엠씨가 운영하는 CSR 플랫폼입니다. 양육자의 심리 상담·정보 제공·커뮤니티를 통해 멘탈 헬스를 케어하는 것이 목표입니다.

3사가 각자 다른 이해관계를 가지고 있어, 우선순위 충돌이 구조적으로 내재된 프로젝트였습니다. 후원사는 신규 기능을 원하고, 운영사는 기술 부채 정비가 필요한 상황이었습니다.

기술 부채가 누적된 상태에서 신규 기능을 추가하면 더 큰 불안정이 발생합니다. 우선순위를 설득하는 것이 PM의 핵심 과제였습니다.

PROBLEM
04

신규 기능 요청 위에 쌓인 기술 부채

플랫폼 구조 자체가 불안정한 상태에서, 후원사 3곳이 각각 다른 기능을 요청하고 있었습니다.

PROBLEM 01
홈 UI 구조 문제
핵심 정보 대신 비핵심 정보가 상단 배치. 사용자가 플랫폼의 목적을 즉시 인지하기 어려운 진입 화면.
PROBLEM 02
보안·인증 불안정
로그인 플로우 및 보안 전반 불안정. DB 스키마 PK/FK 미설정으로 데이터 정합성 저하.
PROBLEM 03
UI/UX 비정형 + 기능 오류
화면별 디자인 일관성 부재. 다수의 기능 오류가 누적되어 사용자 경험 저하.
PROBLEM 04
어드민 전체 개방
권한 분리 없이 어드민 기능 전체 개방. 기능 간 연결 오류로 운영 안정성 위협.
PROBLEM 05
후원사 신규 기능 요청 폭주
3사가 각자 다른 기능을 요청. 기술 부채 위에 신규 기능을 쌓으면 더 큰 불안정 발생 위험.
STRATEGY

신규 기능보다 기술 부채 정비를 먼저

7건의 전문가 인뎁스 인터뷰로 MVP 우선순위를 재정의하고, 3사를 직접 설득해 통과시켰습니다.

DATA
전문가 인뎁스 인터뷰 7건
GA 기반 퍼널형 유입경로 탐색과 PV 분석을 결합. 화면 우선순위를 데이터로 재설정해 감이 아닌 근거로 이야기했습니다.
"가설을 검증하는 질문으로 설계된 인터뷰"
TECH DEBT
인증 흐름·DB 정합성을 기술 부채로 정의
신규 기능 요청 위에 그대로 쌓으면 안 되는 핵심 정비 영역을 명확히 분리. 기술 부채를 설명 가능한 언어로 만들었습니다.
"지금 쌓으면 나중에 더 큰 비용"
PERSUASION
3사 우선순위 충돌 직접 해결
후원사 신규 기능 요청 vs 운영사 기술 부채 정비. 3사를 직접 방문·유선 통화로 만나 우선순위 재정의를 설득했습니다.
"데이터와 사례는 사람을 설득하는 언어"
EXECUTION

핵심 UX와 기술 부채를 동시에 정비

신규 기능보다 안정성을 우선하는 방향으로 4개 영역을 병행 정비했습니다.

ACTION 01
홈 UI/UX 재설계
핵심 정보 노출 우선순위 재배치. 사용자가 플랫폼의 목적을 즉시 인지할 수 있도록 진입 화면을 전면 개편했습니다.
첫 화면에서 플랫폼 목적 즉시 인지
ACTION 02
인증 플로우 안정화
로그인 플로우 및 보안 전반을 정비. 사용자 데이터 안정성을 확보해 플랫폼 신뢰의 기반을 마련했습니다.
사용자 데이터 안정성 확보
ACTION 03
DB 스키마 정비
PK/FK 미설정 문제 해결. 데이터 정합성을 회복해 향후 기능 확장의 기반을 마련했습니다.
데이터 정합성 회복 + 확장 기반 마련
ACTION 04
어드민 권한 분리
전체 개방되어 있던 어드민 기능을 권한별로 분리. 운영 안정성과 보안을 동시에 확보했습니다.
운영 안정성 + 보안 동시 확보

REFLECTION

이 케이스에서 배운 것

LEARNED 01
단기 지표 개선이 곧 제품 정상화를 의미하지는 않았다
KPI 3배 달성은 좋은 결과이지만, 플랫폼의 근본 구조 개선까지는 이르지 못했습니다. 제품 정상화는 한 번의 정비로 끝나지 않는, 지속적 작업입니다. 1차에서 정비한 것이 2차 협업의 기반이 되었습니다.
LEARNED 02
인터뷰 수보다 문제를 검증할 질문 설계가 판단의 정확도를 좌우한다
7건의 인터뷰가 가치 있었던 이유는 수가 많아서가 아니라 각 인터뷰가 가설을 검증하는 질문으로 설계되었기 때문입니다. 데이터는 수집하는 것이 아니라 설계하는 것입니다.
GOOD
  • 전문가 7인 인터뷰로 MVP 우선순위 정의 — 감이 아닌 근거로 설득
  • UI 개편과 기술 부채 정비를 병행해 안정성과 UX 동시 개선
LIMIT
  • 푸시 알림 등 리텐션 장치 부재로 재방문 유도 어려움
  • 플랫폼의 근본적 구조 개선까지는 1차에서 이르지 못함

META REFLECTION

맘스케어가 가르쳐준 것

LEARNED 01
PM은 기술 부채를 이해관계자가 이해하는 언어로 번역해야 한다
기술 부채를 "당장 쌓으면 나중에 더 큰 비용"이라는 언어로 전달하지 않으면, 후원사는 지금 원하는 기능을 왜 못 만드는지 이해할 수 없습니다. 설득은 데이터를 가진다는 게 아니라 상대방이 납득할 수 있는 방식으로 전달하는 것입니다.
LEARNED 02
신뢰는 계약이 끝난 다음에 남는 것이다
1차 계약 종료 후 1년이 지나 재협업을 요청받은 것은, 그 기간 동안의 신뢰가 유지되었기 때문입니다. 프로젝트 기간 안의 성과뿐 아니라, 종료 후에도 신뢰 관계가 지속되는 방식으로 일하는 것이 PM의 자산입니다.
